# Innoliva client onboarding — design

**Date:** 2026-07-01
**Status:** Approved (pending spec review)

## Goal

Onboard a new client, **Innoliva** (Super Intensive Olive only), into IrrigAI:
6 polos and 77 sectors, each backed by a MyIrrigation VWC soil-moisture probe.
Device discovery is already complete — the serial→`external_id` mapping lives in
`docs/innoliva_device_mapping.csv` (see the `innoliva-onboarding` memory and
`backend/scripts/innoliva_discover.py`).

## Decisions (locked)

- **Entity mapping:** 6 Farms, one per polo (each polo = its MyIrrigation
  project). A single new client `User` owns all 6.
- **Conceição weather:** forecast-only (no iMetos station; project 170's
  `/weather_forecast/detailed` needs no device).
- **Crop profile:** reuse the system-default `olive` `CropProfileTemplate`;
  per-sector `Calibração AI` refines CC/PMP later from each VWC probe.
- **Owner account:** new non-admin client `User`, email `innoliva@irrigai.pt`.
  Password supplied out-of-band and passed to the onboarding script via an env
  var (`INNOLIVA_OWNER_PASSWORD`); only the bcrypt hash is stored. The plaintext
  is never committed.

## Entity structure

```
User: Innoliva (role=grower, non-admin)   email=innoliva@irrigai.pt
 ├─ Farm: Polo de Conceição   project 170, weather_device_id=NULL (forecast-only)
 │   └─ Plot 'Olival' → 13 sectors
 ├─ Farm: Polo de Covadonga   project 167, iMetos device 574
 │   └─ Plot 'Olival' → 17 sectors
 ├─ Farm: Polo de Fátima      project 168, iMetos device 590
 │   └─ Plot 'Olival' → 15 sectors
 ├─ Farm: Polo de Guadalupe   project 171, iMetos device 571
 │   └─ Plot 'Olival' → 2 sectors
 ├─ Farm: Polo de Rocio       project 554, iMetos device 573
 │   └─ Plot 'Olival' → 17 sectors
 └─ Farm: Polo do Carmo       project 169, iMetos device 572
     └─ Plot 'Olival' → 13 sectors
```

Per sector: `crop_type="olive"`; `variety` set when the name implies it
(Arbequina / Cobrançosa / Picoal); one `SectorCropProfile` copied from the
`olive` template; one `Probe` with `external_id` from the CSV, `unit=vwc_m3m3`,
`provider="myirrigation"`. `ProbeDepth` rows are created by ingestion from the
first readings (sensor depths vary 5–60 cm per device — not hardcoded).

## Schema change

`FarmCredentials` currently stores encrypted `username`/`password`/`client_id`/
`client_secret` plus a plaintext `weather_device_id`, but **no `project_id`**.
Add a nullable `project_id: str` column (plaintext String(50); not a secret) via
an autogenerated Alembic migration (`make makemigration`, verify the diff).

`adapters/factory._get_myirrigation` must pass `creds.project_id` through to the
adapter for both probe and weather providers (today probe construction does not
pass a per-farm project_id; weather already passes `creds.weather_device_id`).

## Provider / token contention (top risk)

All 6 Innoliva farms share ONE MyIrrigation login (`username=innolivaapi`,
`client_id=0SDX1SAOZA0Y6BTG16Y1`). This is the first time multiple farms share a
single `client_id` — Esporão and Conqueiros use distinct accounts. The prior
"406 Client Signature Invalid" outages (see memories `flowmeter-406-token-invalidation`,
`flowmeter-outage-client-signature`) were caused by **competing logins for the
same `client_id` invalidating each other's cached JWTs**.

Today `adapters/factory` caches a `MyIrrigationAdapter` **instance per credential
+ project + device tuple**, so the 6 farms would get 6 instances, each
authenticating independently against the same `client_id` — recreating exactly
the token-poisoning condition.

**Mitigation (in scope):** cache the adapter/token per `(base_url, username,
password, client_id, client_secret)` only, and carry `project_id` /
`weather_device_id` per **request** rather than baking them into instance state,
so all 6 farms reuse a single shared token. Concretely:

- `fetch_readings` already needs only `device_id` (parsed from `external_id`) —
  no change needed there.
- Weather calls (`fetch_forecast`, `fetch_observations`, `fetch_et0`,
  `_get_weather_project_id`) currently read `self._project_id` /
  `self._weather_device_id`. Add optional `project_id` / `weather_device_id`
  parameters to these methods (falling back to instance state for the existing
  single-project Esporão/Conqueiros callers), and have the farm-aware ingestion
  path pass the polo's values per call.
- Factory: key `_myirrigation_cache` on the credential tuple only; do not include
  project/device in the key.

This preserves existing single-account behaviour (Esporão/Conqueiros unchanged —
they keep passing project/device at construction and have their own client_ids)
while making the shared-account case safe. If per-call threading proves too
invasive, the fallback is to first validate — against the live API — whether
MyIrrigation permits concurrent tokens per `client_id`; but the shared-token
design is the safe default and is what we build.

## Creation mechanism

A single idempotent, re-runnable script `backend/scripts/onboard_innoliva.py`:

1. Reads `docs/innoliva_device_mapping.csv` (the source of truth for the 77
   sector→device rows) and the polo→project/iMetos table above.
2. Get-or-creates, by natural key, in order: the client `User` (by email) →
   6 `Farm`s (by name) → 6 `Plot`s (by farm+name) → 77 `Sector`s (by plot+name)
   → 77 `Probe`s (by `external_id`) → `SectorCropProfile` per sector (copied from
   the `olive` template) → 6 `FarmCredentials` (by farm; encrypted username/
   password/client_id/client_secret from `INNOLIVA_*` env vars, plus per-polo
   `project_id` + `weather_device_id`).
3. Prints a summary and is safe to re-run (no duplicates).

Credentials and the owner password come from env vars at run time
(`INNOLIVA_USERNAME/PASSWORD/CLIENT_ID/CLIENT_SECRET`, `INNOLIVA_OWNER_PASSWORD`);
nothing secret is written to the repo. This mirrors the get-or-create patterns in
`app/seed.py` but is a standalone onboarding script, not part of `seed`.

## Rollout

1. Land the schema change + factory/adapter per-call weather change with tests.
2. Run `onboard_innoliva.py` in **dev** first; verify counts and a sample
   ingestion.
3. Migrate + run on **prod** (`make migrate`, then the script), rebuild
   backend/worker if adapter/factory code changed (`up -d --build backend worker`
   — no source mount in prod).

## Verification / success criteria

- Counts: 1 client User, 6 Farms, 6 Plots, 77 Sectors, 77 Probes, 77
  SectorCropProfiles, 6 FarmCredentials.
- A manual ingestion cycle for 2–3 sample sectors (across ≥2 polos) lands VWC
  readings (`unit=vwc_m3m3`), with `ProbeDepth`s created.
- Weather resolves per-polo (correct project_id); Conceição runs forecast-only
  without error.
- No 406 "Client Signature Invalid" storm across the 6 farms sharing the login
  (validates the shared-token mitigation) — check over one full scheduler cycle.
- Login as `innoliva@irrigai.pt` sees exactly the 6 polo-farms (ownership scope).

## Out of scope

- UI to edit probe `external_id` (a device rename still needs a DB update).
- Super-intensive-specific agronomic template (Kc/density) — deferred; the
  generic `olive` template + calibration is used for now.
- Backfill of historical readings beyond the adapter's adaptive lookback.
