# Flowmeter Dashboard — Herdade dos Conqueiros

**Date:** 2026-05-22  
**Status:** Approved  
**Scope:** Backend models + ingestion service + API endpoints + frontend tab dashboard  

---

## Overview

Add a real-time irrigation consumption dashboard fed by 49 remote flowmeters (caudalímetros) installed across Conqueiros's almond and olive sectors. Data is fetched from the existing MyIrrigation API using the farm's existing credentials. The dashboard lives in a completely separate tab from the existing recommendation view — no existing functionality is touched.

---

## What is NOT changing

- The existing recommendation dashboard, sector cards, probe charts, and water event UI are untouched.
- The `irrigation_event` table, probe ingestion pipeline, and recommendation engine are untouched.
- The MyIrrigation adapter's existing probe-fetching logic is untouched.
- All other farms (Esporão, ADL) are completely unaffected — no tab appears for them.

## Data Models

### `Flowmeter` (`models/flowmeter.py`)

One row per sector that has a physical flowmeter. One-to-one with Sector (a sector has at most one flowmeter).

| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | |
| sector_id | UUID FK → Sector | indexed |
| external_device_id | Integer | MyIrrigation device ID (e.g. 6980) |
| serial_number | String(255) nullable | e.g. "E62300554" |
| name | String(255) | e.g. "Caudalímetro S01 Amendoal" |
| is_active | Boolean | default True |
| last_reading_at | DateTime(tz) nullable | updated after each ingest |

### `FlowmeterReading` (`models/flowmeter_reading.py`)

Time-series table. Converted to a TimescaleDB hypertable on the same column used for `probe_reading`.

| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | |
| flowmeter_id | UUID FK → Flowmeter | indexed |
| timestamp | DateTime(tz) | composite index with flowmeter_id |
| value_m3_ha | Float | m³/ha per 15-min interval (rate, not cumulative) |

Deduplication via `INSERT … ON CONFLICT (flowmeter_id, timestamp) DO NOTHING`.

### `IrrigationEventDetected` (`models/irrigation_event_detected.py`)

Auto-detected irrigation events derived from flowmeter readings. Separate from the existing `irrigation_event` table (which stores manually-logged and recommendation-accepted events).

| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | |
| flowmeter_id | UUID FK → Flowmeter | |
| sector_id | UUID FK → Sector | denormalised for query convenience |
| start_time | DateTime(tz) | first non-zero reading |
| end_time | DateTime(tz) | last non-zero reading |
| duration_minutes | Float | computed |
| total_m3_ha | Float | sum of all readings in the event |
| peak_m3_ha | Float | max reading (quality assurance) |
| num_readings | Integer | number of non-zero readings |
| date | Date | for daily grouping |

All three models are registered in `models/__init__.py` and picked up by Alembic autogenerate.

## Ingestion Architecture (Option A)

Reuse `MyIrrigationAdapter` directly — no new abstract classes or factory changes.

### `parse_flowmeter_data()` in `adapters/myirrigation.py`

A new module-level function alongside the existing `_parse_device_readings`:

```python
def parse_flowmeter_data(
    raw: dict,
    device_id: int,
) -> list[tuple[datetime, float]]:
    """
    Extract (timestamp, value_m3_ha) pairs from a MyIrrigation device data response.
    Looks for sensor_type == "Water Meter" (sensor id "0_0_0").
    Returns [] if the sensor is absent or the response is malformed.
    """
```

Uses the existing `_unix_ms_to_datetime` helper. Ignores all other sensor types.

### `IrrigationEventDetector` in `services/flowmeter_ingestion.py`

Pure, stateless, fully unit-testable:

```python
@dataclass
class DetectedEvent:
    start_time: datetime
    end_time: datetime
    duration_minutes: float
    total_m3_ha: float
    peak_m3_ha: float
    num_readings: int

class IrrigationEventDetector:
    def detect_events(
        self,
        readings: list[tuple[datetime, float]],
        threshold_m3_ha: float = 0.5,
    ) -> list[DetectedEvent]:
        """
        Scan chronologically. Open an event when value > threshold;
        close it when value drops to ≤ threshold. Discard events with
        fewer than 2 readings (noise). Compute total, peak, duration.
        """
```

### `FlowmeterIngestionService` in `services/flowmeter_ingestion.py`

```python
class FlowmeterIngestionService:
    async def ingest_farm(self, farm_id: str, db: AsyncSession) -> dict:
        """
        1. Load farm + FarmCredentials (eagerly).
        2. Get adapter via get_probe_provider(settings, farm=farm).
        3. If isinstance(adapter, MockProbeProvider): log debug, return {}.
        4. Cast to MyIrrigationAdapter.
        5. For each active Flowmeter in the farm:
           a. Compute since = adaptive_since(flowmeter.last_reading_at, 2h, now)
           b. Call ingest_device(flowmeter, since, until, adapter, db)
        6. Commit.
        Returns {"flowmeters_ingested": N, "readings_inserted": N, "events_detected": N}
        """

    async def ingest_device(
        self,
        flowmeter: Flowmeter,
        since: datetime,
        until: datetime,
        adapter: MyIrrigationAdapter,
        db: AsyncSession,
    ) -> int:
        """
        1. POST /data/devices/{external_device_id}/data with date range.
        2. parse_flowmeter_data(raw, device_id) → list[tuple[datetime, float]]
        3. Bulk insert with ON CONFLICT DO NOTHING.
        4. Run IrrigationEventDetector on the new readings window.
        5. Upsert detected events (match on flowmeter_id + start_time).
        6. Update flowmeter.last_reading_at.
        Returns count of new readings inserted.
        """
```

The `_adaptive_since` helper is imported from `services/ingestion.py` (already public).

## API Endpoints

New file: `api/v1/flowmeter.py`. Registered in `router.py` — no existing routes touched.

### `GET /sectors/{sector_id}/flowmeter`
Returns `FlowmeterOut` (metadata). 404 if no flowmeter exists for this sector.

### `GET /sectors/{sector_id}/flowmeter/readings`
Query params: `since`, `until`, `interval` (`15m` default | `1h` | `1d`)

Downsampling:
- `15m`: raw rows ordered by timestamp
- `1h`: `date_trunc('hour', timestamp)` + `SUM(value_m3_ha)`
- `1d`: `date_trunc('day', timestamp)` + `SUM(value_m3_ha)`

Response:
```json
{
  "flowmeter_id": "...",
  "sector_name": "S01",
  "crop": "almond",
  "unit": "m3/ha",
  "interval": "15m",
  "readings": [{"timestamp": "...", "value": 1.8}, ...]
}
```

### `GET /sectors/{sector_id}/flowmeter/events`
Query params: `since`, `until`

Returns detected events for the sector plus a summary (total events, total m³/ha, avg m³/ha per event, period days).

### `GET /farms/{farm_id}/flowmeter-dashboard`
Query param: `period` (`7d` | `30d` | `season`)

`season`: computes window as `MIN(flowmeter_reading.timestamp)` for the farm → today.

Returns aggregated data for all 49 sectors including:
- Per-sector: `total_m3_ha`, `num_events`, `last_irrigation`, `last_event_m3_ha`, `daily_breakdown[]`
- By-crop aggregation: `almond` and `olive` totals
- Farm-level total

The `daily_breakdown` always has exactly N days (N = period in days), zero-filled for days with no irrigation.

## Frontend

### Architecture

- **No changes** to `app/farms/[farmId]/page.tsx` content or existing components.
- New page: `app/farms/[farmId]/caudalimetros/page.tsx` — the flowmeter dashboard.
- Tab bar rendered in `page.tsx` via a new `FarmTabBar` component. Shown only when `DashboardResponse.has_flowmeters` is true (loaded as part of the existing `useFarmDashboard` hook — no extra API call).
- The tab bar uses URL-based navigation: the "Recomendações" tab links to `/farms/[farmId]`, the "Caudalímetros" tab links to `/farms/[farmId]/caudalimetros`.

### New components

| Component | Purpose |
|-----------|---------|
| `FarmTabBar` | Tab strip (Recomendações / Caudalímetros), only for farms with flowmeters |
| `FlowmeterDashboard` | Main dashboard container — period selector, filter bar, sector table |
| `FlowmeterSectorTable` | Sortable sector table with status dots, sparklines, crop group dividers |
| `FlowmeterSectorRow` | Single row; click expands `FlowmeterSectorDetail` inline |
| `FlowmeterSectorDetail` | Expanded detail: bar chart (Recharts) + events list |
| `FlowmeterSparkline` | Mini 7-bar chart for the table's last column |
| `FlowmeterInfoCard` | Cross-link card on the existing sector detail page |

### Tab bar behaviour

- Appears only when the farm dashboard API returns `has_flowmeters: true` (new field added to `DashboardResponse`).
- On the existing farm page (`/farms/[farmId]`): "Recomendações" tab is active.
- On `/farms/[farmId]/caudalimetros`: "Caudalímetros" tab is active.
- For Esporão and all other farms: `has_flowmeters` is false → no tab bar, page looks exactly as before.

### Dashboard layout

1. **Summary bar**: total m³/ha for the period · by-crop breakdown · period selector (7d / 30d / Campanha)
2. **Filter row**: Cultura filter (Todos / Amendoal / Olival) · Sort dropdown (Setor / Última rega / Total / Nº regas)
3. **Sector table**: one row per sector, sorted by sector name by default, with crop group dividers
4. **Sector row columns**: status dot · Setor · Cultura · Último evento · Última dotação (m³/ha) · Total período · Nº regas · Sparkline 7d

### Status dot colour rules

| Condition | Colour |
|-----------|--------|
| Last irrigation today | Green |
| Last irrigation 1–3 days ago | Green |
| Last irrigation 4–7 days ago | Amber |
| Last irrigation > 7 days ago | Red |
| No data at all | Grey (row dimmed) |

### Sector detail (inline expand on row click)

- Bar chart (Recharts `<BarChart>`) of 15-min readings for the selected period, with event blocks appearing as tall olive-green bars
- When the sector also has a probe: a `<Line>` overlay on a second Y-axis showing VWC% from the probe (using the existing `probesApi.readings()` call)
- Detected events table: Início · Duração · Dotação (m³/ha) · Pico (m³/ha)

### Cross-link card (`FlowmeterInfoCard`)

Added at the bottom of `app/farms/[farmId]/sectors/[sectorId]/page.tsx`. Renders only if the sector has a flowmeter (checked via `sectorsApi.getFlowmeter(sectorId)` — returns null/404 for sectors without one).

Content: "Caudalímetro · Última rega DD/MM · XX.X m³/ha" + "Ver histórico completo →" link to `/farms/[farmId]/caudalimetros`.

### New API client methods (`lib/api.ts`)

```typescript
export const flowmeterApi = {
  getSector: (sectorId: string) =>
    get<FlowmeterOut>(`/sectors/${sectorId}/flowmeter`),
  readings: (sectorId: string, params: { since?: string; until?: string; interval?: string }) =>
    get<FlowmeterReadingsResponse>(`/sectors/${sectorId}/flowmeter/readings?...`),
  events: (sectorId: string, params: { since?: string; until?: string }) =>
    get<FlowmeterEventsResponse>(`/sectors/${sectorId}/flowmeter/events?...`),
  dashboard: (farmId: string, period: "7d" | "30d" | "season") =>
    get<FlowmeterDashboardResponse>(`/farms/${farmId}/flowmeter-dashboard?period=${period}`),
};
```

New TypeScript types added to `types/index.ts`: `FlowmeterOut`, `FlowmeterReadingsResponse`, `FlowmeterEventsResponse`, `FlowmeterDashboardResponse`.

## `has_flowmeters` flag on farm dashboard

One small additive change to `GET /farms/{farm_id}/dashboard`: adds `has_flowmeters: bool` to `DashboardResponse`. The backend computes this with a single `EXISTS` query against the `flowmeter` table. The frontend uses it to conditionally render `FarmTabBar`.

This is the only change to any existing endpoint.
