# Per-plot weather — design

**Date:** 2026-07-03
**Status:** Approved (pending spec review)

## Goal

Let a single Farm draw weather **per Plot** — each plot can have its own
MyIrrigation project + iMetos station — while farms that don't configure it keep
today's single farm-level weather. This unblocks **Innoliva**: one Farm
"Innoliva", 6 Plots (the polos), each polo's sectors using that polo's own iMetos
for ET₀; Conceição (no iMetos) → forecast-only. Dashboard shows one "Innoliva";
the farm page shows the 6 polos as the existing plot tabs.

Context: Innoliva is already modelled as one Farm + 6 Plots + 77 olive sectors
(commit `0cefbb0`; onboarding `backend/scripts/onboard_innoliva.py`,
mapping `docs/innoliva_device_mapping.csv`). Probes are project-agnostic (reads
use only `device_id`), so only **weather** needs the per-plot dimension.

## Background — current weather flow (all farm-scoped)

- `weather_observation` / `weather_forecast` tables are keyed by `farm_id` only.
- Ingestion `services/ingestion.py` fetches once per farm (farm's
  `FarmCredentials` project_id + weather_device_id) and stores rows with
  `farm_id`.
- Engine `engine/pipeline.py` `build_weather_context(farm_id, db)` reads by
  `farm_id`; the recommendation pipeline builds it per farm and shares it across
  all the farm's sectors.

## Design

### 1. Schema (autogenerated migration)
- `plot`: add `myirrigation_project_id: str | None` (String(50)) and
  `weather_device_id: str | None` (String(50)). Both null ⇒ "this plot uses the
  farm's weather" (default; unchanged behaviour).
- `weather_observation` + `weather_forecast`: add `plot_id: str | None`
  (UUID FK → `plot.id`, nullable, indexed). **Null `plot_id` = farm-level
  weather** (all existing rows and all existing farms stay farm-level). Keep the
  existing `farm_id` column (a plot-scoped row still records its farm).
- Update the composite indexes to include `plot_id`
  (`(farm_id, plot_id, timestamp)` / `(farm_id, plot_id, forecast_date)`).

### 2. Adapter (`adapters/myirrigation.py`)
`fetch_forecast`, `fetch_observations`, `fetch_et0`, and `_get_weather_project_id`
gain optional `project_id` / `weather_device_id` params that **override instance
state for that call** (fall back to `self._project_id` / `self._weather_device_id`
when not passed — existing single-project callers unchanged). This lets one
shared adapter instance serve several plots without per-plot adapter instances;
the same 406→re-auth self-heal covers the shared `client_id`.

### 3. Ingestion (`services/ingestion.py`)
- Add a per-plot path in the farm weather step: for each `Plot` of the farm that
  has `weather_device_id` (and/or `myirrigation_project_id`) set, fetch
  observations + forecast using that plot's project/device and store rows with
  `plot_id` set.
- Farms with **no** plot-level weather config keep the existing single farm-level
  fetch (rows with `plot_id = NULL`). A farm may mix: plots without config fall
  back to farm-level.
- Plots whose `weather_device_id` is null but `myirrigation_project_id` is set
  (e.g. Conceição) get **forecast only** (project forecast endpoint needs no
  device); observations/ET₀ from a device are simply empty for them.
- Dedup/backfill logic must key on `(farm_id, plot_id, timestamp)` so plot and
  farm series don't collide.

### 4. Engine (`engine/pipeline.py`)
- `build_weather_context` resolves weather for a sector by its plot: prefer rows
  matching the sector's `plot_id`; if none exist, fall back to farm-level rows
  (`plot_id IS NULL`). Signature gains an optional `plot_id` (farm-level callers
  pass none → current behaviour).
- Confirm where the pipeline builds weather context (today once per `farm_id` at
  `pipeline.py:461`) and make it resolve per the sector's plot. If it's built
  once per farm and reused, change it to build per plot (cache per plot_id within
  a run to avoid refetching). **This is the only change on the recommendation
  path — it must preserve existing farms exactly (verified by tests on a
  farm-level farm).**

### 5. Onboarding (`scripts/onboard_innoliva.py`)
- Set each polo-plot's `myirrigation_project_id` + `weather_device_id` from the
  polo→iMetos map: Covadonga→167/574, Fátima→168/590, Rocio→554/573,
  Guadalupe→171/571, Carmo→169/572, **Conceição→170/None** (forecast-only).
  (This is `POLO_META`, already in the script.)
- The Innoliva **farm-level** `FarmCredentials` weather can be left null (every
  plot overrides); the `INNOLIVA_WEATHER_*` env becomes optional.

## Backward compatibility (hard requirement)

Esporão, Conqueiros, ADL configure no plot-level weather → every weather row
stays `plot_id = NULL` → ingestion and the engine behave exactly as today. The
migration is additive (nullable columns). This must be covered by a test that
builds a farm-level weather context and asserts unchanged behaviour.

## Testing

- Unit: adapter per-call `project_id`/`weather_device_id` override (falls back to
  instance state when omitted).
- Unit/integration: `build_weather_context` returns plot-scoped rows when present,
  falls back to farm-level when absent (both cases).
- Ingestion: a farm with 2 plots (one with weather config, one without) stores
  plot-scoped rows for the first and farm-level for the second; dedup keyed on
  `(farm_id, plot_id, timestamp)`.
- Regression: an existing farm-level farm's weather context is byte-for-byte
  unchanged.
- CI uses mock providers; live per-plot fetch is validated in dev against
  MyIrrigation.

## Rollout

1. Land schema + adapter + ingestion + engine + onboarding with tests; merge to
   `main`.
2. Prod: `git pull` → migrate → rebuild backend+worker → run onboarding (per-plot
   weather config set) → verify each polo ingests from its own iMetos, Conceição
   forecast-only, and Esporão/Conqueiros/ADL weather unchanged. Watch one
   scheduler cycle for the 406 rate.

## Out of scope

- UI to edit plot weather config (set via onboarding / DB for now).
- Backfill of historical per-plot weather beyond adaptive lookback.
- The pre-existing `auto_calibration.analyze_sector` `"moisture"` vs
  `"soil_moisture"` mismatch (tracked separately).
