# Per-plot Weather Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let a single Farm draw weather per Plot (each plot = its own MyIrrigation project + iMetos), with farm-level fallback so existing farms are unchanged. Unblocks Innoliva per-polo weather.

**Architecture:** Additive schema (`plot` weather columns + nullable `plot_id` on the two weather tables) → adapter takes per-call project/device → ingestion fetches per-plot → engine `build_weather_context` resolves by the sector's plot with farm-level fallback → onboarding sets each polo-plot's weather.

**Tech Stack:** Python 3.12, SQLAlchemy 2.0 async, Alembic, pytest-asyncio, httpx, Docker Compose.

## Global Constraints

- Python: ruff line-length 100, target py312.
- Migrations are AUTOGENERATED (`docker compose exec backend alembic revision --autogenerate -m "…"`), verify the diff, never hand-write.
- Tests + migrations run inside Docker: `docker compose exec -T backend …`. Stack is running.
- **Backward-compat is a hard requirement:** `plot_id IS NULL` = farm-level weather; farms with no plot weather config (Esporão, Conqueiros, ADL) must behave byte-for-byte as today. Every new column is nullable.
- Never hardcode secrets.
- Polo→(project_id, iMetos device): Conceição→(170, None), Covadonga→(167,574), Fátima→(168,590), Guadalupe→(171,571), Rocio→(554,573), Carmo→(169,572) — this is `POLO_META` in `backend/scripts/onboard_innoliva.py`.
