Follow-up to `docs/ci-handoff-2026-06-15.md`.

## Summary

Codex handled the two CI issues from the handoff:

1. `alembic check` schema drift.
2. Backend test pollution around probe ingestion rows.

I also fixed the two MyIrrigation adapter assertion drifts called out in the
handoff.

## What Changed

### Schema Drift

Direction chosen: migrations/prod are the source of truth.

I kept the production composite/performance indexes and updated SQLAlchemy model
metadata to match the migration history. This avoids a generated migration that
would drop useful production indexes.

Model metadata now declares existing migration-created indexes:

- `ix_alert_is_active`
- `ix_flowmeter_external_device_id`
- `ix_flowmeter_reading_fm_ts`
- `ix_irrigation_event_sector_start`
- `ix_irrigation_event_detected_fm_start`
- `ix_irrigation_event_detected_sector_start`
- `ix_provider_ingestion_run_started_at`
- `ix_recommendation_generated_at`
- `ix_weather_forecast_farm_date`
- `ix_weather_obs_farm_timestamp`

I removed model-only column comments that were not represented in migrations,
so Alembic no longer wants a comments-only migration.

TimescaleDB creates `probe_reading_timestamp_idx` implicitly for the hypertable.
That is not app-owned schema, so `backend/alembic/env.py` now filters that
reflected index from autogenerate.

### Backend Test Isolation

Added an autouse cleanup fixture in `backend/tests/conftest.py` for committed
rows that tests create across sessions/requests.

The cleanup intentionally preserves normal seeded probe readings. The ingestion
tests now use a future-dated fixed window (`2099-01-01`), and cleanup removes
only future-dated `probe_reading` rows plus telemetry/status rows generated by
tests. This isolates the CI failure without breaking seeded-data API tests.

### MyIrrigation Assertion Drift

`fetch_forecast` parsing now accepts:

- current keyed response shape: `{"values": {"YYYY-MM-DD": {...}}}`
- older/list test fixture shape: `[{date: ...}, ...]`

Watermark sentinel handling is now explicit in tests:

- ordinary negative readings like `-5` are included for ingestion quality
  flagging
- MyIrrigation no-data sentinels `-252` / `253` are skipped
