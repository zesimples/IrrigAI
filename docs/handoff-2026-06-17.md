## Next focus: APScheduler not auto-ingesting on prod

**Symptom:** Production data only updates when ingestion is run **manually**
(`scripts/ingest_all_farms.py`, `scripts/ingest_flowmeters.py`). The scheduled jobs are
not keeping probe/flowmeter/weather data current — data had frozen at 15/06 and only
caught up after manual runs on 17/06.

**Important:** this is now a *scheduler* problem, **not** the MyIrrigation 406 — that
signature issue is **resolved upstream** (manual ingestion on 17/06 pulled cleanly:
flowmeters `readings_inserted: 2182, events_detected: 44`, `last_reading_at` → 2026-06-17;
probes ingest fine for all 3 farms). So the API works; the worker just isn't firing/finishing
the scheduled jobs.

**Architecture (where to look):**
- `backend/app/services/scheduler.py` — APScheduler jobs: `_run_data_ingestion`
  (id `data_ingestion`, ~15 min) and `_run_flowmeter_ingestion` (id `flowmeter_ingestion`,
  ~20 min). Both wrap work in a Redis `JobLock` (`backend/app/services/job_lock.py`,
  TTL 900s / 1200s) and increment `scheduler_job_runs_total{job,result}`.
- Worker container runs `python -m app.worker`, which should start the scheduler.

**First diagnostics (run on prod):**
1. Are the jobs firing at all? `docker compose logs --since 3h worker | grep -iE "Scheduler|ingestion|apscheduler|lock"` — expect "Scheduler: data ingestion at …" lines every ~15 min. If absent → scheduler isn't ticking (boot crash / not started / thread died).
2. Is the worker healthy and did the scheduler start? `docker compose logs --since 24h worker | head -50` (look for APScheduler start + any traceback on boot).
3. Stuck Redis lock? A lock acquired and never released makes every run "skipped":
   `docker compose exec redis redis-cli --scan --pattern '*lock*'` then `TTL <key>`.
   (TTL should be ≤ the job TTL; a `-1` no-expiry lock would wedge it.)
4. Metric check (if Prometheus is up): `scheduler_job_runs_total` — are results `skipped`
   vs `success` vs `failure`?

**Leading hypotheses:** (a) APScheduler never started / died in the worker (boot error or
the worker is running stale code from a baked image — prod bakes code, needs
`docker compose up -d --build worker`, not just `restart`); (b) a stuck Redis job lock from
a crash during the 406 outage (jobs were failing then); (c) a job raising and killing the
scheduler thread; (d) interval/timezone/misfire-grace misconfig.

**Quick interim mitigation:** a cron entry calling the two manual ingest scripts every
15–20 min until the scheduler is fixed.
