# AI P3–P6 handoff — 2026-07-23

## Scope

This batch completes the implementation work planned for AI phases P3–P6. The
deterministic engine remains authoritative: the LLM explains outcomes and proposes
actions, but it does not calculate irrigation decisions, calibration limits, or
outcome residuals.

## P3 — Memory and chat

- Added user-scoped, farm-scoped persisted chat conversations and messages.
- Added conversation list, detail, delete, resume, and SSE chat endpoints.
- Added sector field observations with optional structured data, expiry, and
  verification state. Active observations feed the canonical V2 AI context and are
  explicitly marked as user observations and verified/unverified.
- Field notes entered on the sector analysis card are retained for seven days.
- Added read-only chat tools for outcomes, calibration state, recommendation
  history, flowmeter execution, and stress projection. All tool calls retain farm
  and sector ownership checks.

## P4 — Cost and robustness

- Added login, registration, and LLM endpoint rate limits.
- Added bounded chat/history/note/window inputs and a Redis-backed daily per-user
  LLM ceiling. The quota fails open when Redis is unavailable and is bypassed by
  the mock provider.
- Added OpenAI request timeout and retry settings.
- Structured and conversational LLM failures now produce an explicit degraded
  response, log the error, and increment a Prometheus counter. The UI labels these
  responses as contingencies instead of presenting them as normal low-confidence
  output.
- Farm summaries are cached in Redis by a stable digest of their canonical context.
- Farm-wide context construction now uses bounded aggregate queries instead of
  serially building a full context for every sector.
- Removed pretty-printed JSON from LLM prompts.
- Added thumbs feedback persistence and a bounded-label feedback metric.

## P5 — New surfaces

- Added an `Eficácia da rega` sector tab with:
  - recent deterministic recommendation outcomes;
  - recommended/applied dose and residual;
  - qualitative probe response by depth;
  - calibration candidate history and apply action;
  - probe-ingestion diagnostics;
  - 72-hour change analysis;
  - an AI explanation of measured irrigation effectiveness.
- Recommendation outcome evaluation now stores per-depth probe response as
  `increase`, `stable`, or `decrease`. Raw VWC values are not rendered to the
  farmer or delegated to the LLM for calibration.
- Added AI alert explanations to the alerts page.

## P6 — Model routing

- Added independent routing settings for conversational, structured, and farm
  summary workloads:
  - `OPENAI_MODEL_CHAT`
  - `OPENAI_MODEL_STRUCTURED`
  - `OPENAI_MODEL_SUMMARY`
- Blank overrides inherit `OPENAI_MODEL`.
- Every structured surface is tagged in production and in the live golden-set
  runner, so candidate routing changes can be evaluated per surface.
- No model promotion is made in this batch. Run the live golden set against each
  candidate and only set production overrides when the complete set passes.

## Production deployment

The new tables must exist before the new backend starts serving traffic.

On the current Caddy host:

```bash
git pull --ff-only origin main

docker compose \
  -f docker-compose.yml \
  -f docker-compose.prod.yml \
  -f docker-compose.caddy.yml \
  build backend worker frontend

docker compose \
  -f docker-compose.yml \
  -f docker-compose.prod.yml \
  -f docker-compose.caddy.yml \
  run --rm backend alembic upgrade head

docker compose \
  -f docker-compose.yml \
  -f docker-compose.prod.yml \
  -f docker-compose.caddy.yml \
  up -d --no-deps backend worker frontend
```

`docker-compose.caddy.yml` so Caddy can reach loopback ports 8000 and 3000.
